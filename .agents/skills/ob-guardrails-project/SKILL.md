# Non-Negotiable Guardrails

## Architecture

### GR-ARCH-001: Electron Process Ownership

The application uses a strict three-process Electron model:

- `src/main/` owns all network I/O, filesystem access, operating-system integration, process management, SSH operations, and secure persistence.
- `src/preload/` is a thin typed `contextBridge`.
- `src/renderer/` is sandboxed and contains presentation and client-side application logic only.

The renderer must not perform direct network, filesystem, process, or Node.js operations.

Evidence:

- `ARCHITECTURE.md` §2 and §9
- `contextIsolation: true`
- `nodeIntegration: false`

### GR-ARCH-002: Main Process Owns HTTP and SSE

All HTTP and SSE communication with loop-task daemons runs in the main process.

The daemon does not provide browser CORS headers, and remote daemon communication must work independently of browser restrictions.

The renderer reaches daemon capabilities only through injected services backed by the typed preload bridge.

Never add renderer-side `fetch`, `EventSource`, WebSocket access, or equivalent direct daemon communication.

Evidence:

- `ARCHITECTURE.md` §2 and §14
- `src/main/index.ts`
- `handleApiRequest`
- main-process stream subscription handling

### GR-ARCH-003: Shared IPC Contract

`src/shared/ipc.ts` is the single source of truth for all cross-process contracts.

All request types, response types, bridge interfaces, event payloads, configuration contracts, and translated message types shared between Electron processes must be defined there.

Any new or modified IPC capability must update the shared contract first.

Do not duplicate IPC types in main, preload, renderer, or service modules.

Evidence:

- `ARCHITECTURE.md` §3.3
- `src/shared/ipc.ts`

### GR-ARCH-004: Import Boundaries

Process import boundaries are strict:

- Renderer code may import from `src/shared/` and `src/renderer/src/`.
- Renderer code must not import from `src/main/` or `src/preload/`.
- Preload code may import from `src/shared/` and `src/preload/`.
- Preload code must not import renderer modules.
- Main-process code may import from `src/shared/` and `src/main/`.

Respect the TypeScript project boundaries defined by `tsconfig.web.json` and `tsconfig.node.json`.

### GR-ARCH-005: Renderer Uses Injected Services

React components and hooks must never access the preload bridge, `window.api`, `ipcRenderer`, or Node APIs directly.

Renderer infrastructure accesses Electron capabilities only through injected services.

- Real service implementations may consume the typed preload bridge internally.
- Components and hooks resolve abstractions using `useInject<IXxxService>(cid.IXxxService)`.
- Non-React infrastructure may resolve dependencies through the configured container when hooks are unavailable.
- Browser-only development uses mock service implementations registered in the same dependency-injection container.
- Components must not manually select between real and mock implementations.

The dependency-injection container is configured in the `app/` FSD layer.

Evidence:

- `src/renderer/src/services/container.ts`
- `@inversify-hooks`
- current real and mock service implementations

### GR-ARCH-006: Main Boundary Unwraps API Responses

The loop-task daemon returns response envelopes such as:

```ts
{ ok: true, data: value }
```

and:

```ts
{ ok: false, error: { message: "..." } }
```

The main process must unwrap daemon envelopes into the shared `ApiResponse<T>` contract.

Do not push daemon-specific envelope handling into renderer components, hooks, or UI services.

Evidence:

- `ARCHITECTURE.md` §3.2
- `src/main/index.ts`
- `handleApiRequest`

### GR-ARCH-007: SSH Onboarding Boundary

SSH onboarding may bootstrap loop-task only with explicit user consent.

The SSH onboarding flow must:

- authenticate before making remote changes
- require Node.js 20 or later
- detect the `loop-task` command separately from daemon state
- offer installation and startup when loop-task is missing
- use the existing pinned npm installation path
- bind the daemon to `127.0.0.1`
- verify startup using a bounded readiness check
- surface real remote output when installation or startup fails

Local and direct instances remain user-managed and must never enter the SSH installation path.

Evidence:

- `ARCHITECTURE.md` §3.2 and §4
- `src/main/vm-wizard.ts`
- `src/main/ssh-probe.ts`
- `src/main/ssh-launch.ts`

## Security

### GR-SEC-001: Renderer Sandboxing

Electron renderer sandboxing must remain enabled:

- `contextIsolation: true`
- `nodeIntegration: false`

The renderer has no direct Node.js, filesystem, process, credential, or network access.

Do not weaken these settings to simplify implementation.

### GR-SEC-002: Narrow Trust Boundary

The trust boundary is:

```text
renderer → preload → main
```

The preload exposes only explicitly typed capabilities.

Do not expose generic IPC send methods, unrestricted channel names, arbitrary command execution, arbitrary filesystem access, or raw Electron APIs.

### GR-SEC-003: URL Validation

Main-process network requests may target only validated `http:` and `https:` URLs.

Reject unsupported protocols before making requests.

Do not allow `file:`, `javascript:`, custom protocols, or other unapproved URL schemes through daemon configuration.

Evidence:

- `src/main/index.ts`
- `isAllowedBaseUrl()`

### GR-SEC-004: External Navigation

External links must open in the user's system browser.

In-app navigation to arbitrary external pages must remain denied through the Electron window-open handler.

Evidence:

- `ARCHITECTURE.md` §9
- `setWindowOpenHandler`

### GR-SEC-005: Credential Storage

Secrets must never be stored unencrypted.

Wizard credentials use a dedicated reference vault:

- environments contain only opaque `credentialRefs`
- session tokens and SSH key passphrases are encrypted with Electron `safeStorage`
- encrypted values live in the dedicated credential vault
- no plaintext fallback is permitted
- removing an environment removes its owned vault entries
- credential values and ciphertext must never be added to `Environment`

Existing OpenCode passwords must continue to use the config-store encryption wrapper.

Evidence:

- `src/main/credential-vault.ts`
- `src/main/config-store.ts`
- `ARCHITECTURE.md` §5 and §9

### GR-SEC-006: Secret-Safe Serialization

Serialized or synchronized environment data must use explicit allowlists at every nesting level.

Secret or internal fields must never appear in synchronized output, including:

- `password`
- `wasEncrypted`
- `encryptedAccessToken`
- `encryptedValue`
- `accessToken`
- credential ciphertext
- session tokens
- SSH key passphrases

Fields not included in the serialization allowlist must be dropped.

Evidence:

- `sanitizeEnvironmentForSync`
- `SECRET_FIELD_NAMES`
- `src/main/config-store.ts`

### GR-SEC-007: Main-Process Token Injection

Authorization tokens are attached only in the main process.

The renderer must never receive raw bearer tokens.

`handleApiRequest` and SSE subscription handlers attach:

```text
Authorization: Bearer <token>
```

When a daemon returns `401`:

- remove the invalid session token
- set the environment authentication state to `blocked`
- surface an appropriate translated error

Evidence:

- `src/main/index.ts`
- API and stream authentication handling

## Data and Persistence

### GR-DATA-001: Daemon-Owned Domain Data

The desktop application owns no authoritative loop-task domain data.

Projects, loops, runs, tasks, statuses, and logs are owned by the configured loop-task daemons.

The application may cache transient view state but must not create a competing durable domain store.

### GR-DATA-002: Main-Process Configuration Persistence

Persistent application configuration belongs in the main process.

Use typed `electron-store` persistence in Electron's `userData` directory for:

- environments
- selected environment
- credential references
- session metadata
- other approved application configuration

The renderer accesses configuration only through typed injected services.

### GR-DATA-003: No Renderer Config Persistence

Do not use `localStorage` for real Electron environment or credential configuration.

Browser-only mock mode may use browser storage when necessary for development, but mock persistence must not become the production persistence path.

### GR-DATA-004: Window Bounds

Window bounds are stored in `window-bounds.json` inside Electron's `userData` directory.

Bounds persistence remains debounced to avoid excessive writes.

### GR-DATA-005: SSE Parsing

SSE data must be parsed using the existing standards-compliant parser based on `eventsource-parser`.

Do not reintroduce manual `\n\n` splitting or other hand-written stream framing.

The parser must continue to support:

- chunk-boundary splits
- multiline `data:` fields
- event IDs
- retry fields
- comment lines
- valid WHATWG SSE framing

Evidence:

- `src/main/sse-parser.ts`
- `eventsource-parser`

### GR-DATA-006: Stream Lifecycle

Active SSE subscriptions are tracked using:

```ts
Map<subId, AbortController>
```

Subscriptions must be aborted when:

- the renderer unsubscribes
- the owning window or environment is removed
- the application shuts down
- the connection becomes invalid

Do not leave detached stream readers running.

# File and Module Structure

## GR-STRUCT-001: Small, Cohesive Files

Keep every source file small, cohesive, and single-purpose.

This applies to:

- React components
- hooks
- services
- adapters
- repositories
- stores
- IPC handlers
- Electron lifecycle modules
- SSH modules
- configuration modules
- formatters
- validators
- parsers
- test files
- fixtures
- scripts
- build configuration
- shared contracts

File length is not itself a violation. A large file is a signal to inspect cohesion.

Split a file when it contains concerns that can be independently named, understood, tested, replaced, or changed for different reasons.

Do not split files only to satisfy an arbitrary line-count target.

## GR-STRUCT-002: One Primary Concept per File

Each source file should expose one clear primary responsibility.

A file may contain tightly coupled private helpers, private constants, and private types when they support the same responsibility.

Avoid files that act as collections of unrelated exports.

## GR-STRUCT-003: No God-Files or Dumping Grounds

Do not create generic catch-all files such as:

- `constants.ts`
- `types.ts`
- `config.ts`
- `utils.ts`
- `helpers.ts`
- `common.ts`
- `misc.ts`
- `shared.ts`

unless the file represents one genuinely cohesive domain concept.

Prefer domain-specific names such as:

- `loop-types.ts`
- `environment-config.ts`
- `ssh-validation.ts`
- `wizard-constants.ts`
- `log-formatting.ts`
- `connection-errors.ts`

A file importing many unrelated domains may indicate mixed responsibilities or misplaced orchestration. Evaluate cohesion and dependency direction before splitting.

## GR-STRUCT-004: Separate Orchestration from Logic

Modules that coordinate several dependencies should contain orchestration only.

Extract:

- parsing
- validation
- classification
- mapping
- transformation
- formatting
- decision logic
- state transitions

into small pure modules where practical.

Electron-coupled modules should delegate testable logic to pure functions rather than embedding all behaviour inside IPC handlers or lifecycle callbacks.

## GR-STRUCT-005: Separate Interfaces from Implementations

Renderer service abstractions, real implementations, and mock implementations must remain distinct.

A recommended structure is:

```text
services/
  contracts/
    loop-service.ts
    config-service.ts
    log-service.ts
  real/
    electron-loop-service.ts
    electron-config-service.ts
  mock/
    mock-loop-service.ts
    mock-config-service.ts
  container.ts
  identifiers.ts
```

Do not place interfaces, real implementations, mock implementations, and container registration in one large file.

## GR-STRUCT-006: Group by Domain or Feature

Prefer grouping files around a domain capability rather than technical file type alone.

Good examples:

```text
features/
  environment-onboarding/
    ui/
    model/
    api/
    lib/
```

```text
main/
  ssh/
    ssh-probe.ts
    ssh-launch.ts
    ssh-validation.ts
    ssh-errors.ts
```

Avoid broad global folders that become unrelated collections.

## GR-STRUCT-007: Public APIs for Slices and Modules

When a feature or domain contains several internal files, expose a narrow public API through an `index.ts` only when it improves dependency control.

Do not use barrel files to re-export entire directories indiscriminately.

A public API should expose only what external consumers need.

Internal files should remain private to the slice or module.

## GR-STRUCT-008: Avoid Circular Dependencies

Do not introduce circular imports between:

- Electron processes
- FSD layers
- services
- domain modules
- feature slices
- shared contracts

When two modules require each other, extract the shared contract or pure domain concept into a lower-level module.

Do not resolve circular dependencies through dynamic imports unless the runtime boundary genuinely requires them.

## GR-STRUCT-009: Keep Entry Points Thin

Entry-point files such as these should contain composition and startup wiring only:

- `src/main/index.ts`
- preload entry points
- renderer application bootstrap
- dependency-injection container setup
- Vite or Electron startup configuration

Move business logic, parsing, validation, connection management, and persistence logic into dedicated modules.

## GR-STRUCT-010: Keep Tests Focused

Test files should remain focused on one module or one cohesive behaviour.

Split tests when a file covers unrelated modules, unrelated user flows, or distinct runtime boundaries.

Shared test builders and fixtures must be domain-specific rather than becoming generic dumping grounds.

# Renderer Structure

## GR-FSD-001: Feature-Sliced Design

The renderer follows Feature-Sliced Design:

```text
app → pages → widgets → features → entities → shared
```

Upper layers may import from lower layers only.

Lower layers must not import from upper layers.

Examples:

- `pages` may import `widgets`, `features`, `entities`, and `shared`
- `features` may import `entities` and `shared`
- `entities` may import `shared`
- `shared` must not import domain features or entities

Load `@feature-sliced-design` before making renderer structural decisions.

## GR-FSD-002: Shared Layer Is Truly Cross-Cutting

The `shared/` layer contains only generic, reusable infrastructure such as:

- design tokens
- base UI primitives
- generic hooks
- generic utility functions
- generic infrastructure adapters
- internationalization helpers

Domain-specific logic belongs in `entities/` or `features/`.

Do not move code into `shared/` merely because several files currently use it.

## GR-FSD-003: Features Own User Capabilities

A `feature` represents a user capability or interaction.

Features may contain:

- UI
- feature-specific state
- feature-specific hooks
- feature-specific validation
- feature-specific service coordination

Do not create a feature for generic visual primitives or passive domain representation.

## GR-FSD-004: Entities Own Domain Representation

Entities represent stable domain concepts such as:

- environment
- loop
- project
- task
- run
- connection

Entity slices may contain:

- domain types
- mapping
- display models
- domain UI
- selectors
- pure domain logic

Entity modules must not depend on user-flow-specific feature logic.

## GR-FSD-005: Widgets Compose Larger UI Regions

Widgets compose entities and features into substantial page regions.

Widgets should not become application-wide state containers or repositories for unrelated business logic.

Move reusable domain behaviour into entities or features.

## GR-FSD-006: Pages Coordinate Routes and Screens

Pages coordinate widgets and features for a complete screen.

Pages should remain thin and should not contain low-level reusable domain logic.

## GR-FSD-007: App Layer Owns Application Composition

The `app/` layer owns:

- application bootstrap
- providers
- dependency-injection configuration
- routing
- global error boundaries
- internationalization setup
- global styles
- application-level lifecycle

Features must not configure the global dependency-injection container directly.

- `processor`
- `handler`
- `data`
- `thing`
- `common`

Internationalization keys use dot notation with a domain prefix:



```text

```

Do not change the schema of an existing versioned key without incrementing its version or adding a migration.

# Code Style

## GR-STYLE-001: Strict TypeScript

TypeScript remains strict.

The project enables:

- `strict`
- `noUnusedLocals`
- `noUnusedParameters`
- `forceConsistentCasingInFileNames`

Do not use `any`.

Prefer:

- precise interfaces
- discriminated unions
- generics
- `unknown` with validation
- exhaustive checks
- typed adapters at external boundaries

## GR-STYLE-002: ESM Only

The project uses ESM.

Main and preload imports use `.js` extensions where required by the current bundler and TypeScript configuration.

Do not introduce CommonJS `require`, `module.exports`, or mixed module patterns.

## GR-STYLE-003: Function Components and Hooks

React code uses function components and hooks.

Do not introduce class components.

## GR-STYLE-004: Plain CSS and Design Tokens

Styling uses plain CSS with CSS custom-property design tokens.

Do not introduce:

- Tailwind
- CSS-in-JS
- styled-components
- another CSS framework

without explicit team alignment and an approved architecture change.

Load `@accelint-design-foundation` before styling work.

## GR-STYLE-005: Internationalized User-Facing Copy

All user-facing text must use internationalization keys.

Do not hardcode visible product copy in:

- React components
- hooks
- services
- main-process errors
- dialogs
- notifications
- onboarding flows

The main process creates translated message objects through `msg(key, params)`.

The renderer uses `react-intl`, `formatMessage`, or `translateMessage()` as appropriate.

## GR-STYLE-006: Comments Explain Intent

Code should be understandable through naming, structure, types, and small cohesive modules.

Comments should explain:

- non-obvious reasoning
- architectural constraints
- security decisions
- external-system limitations
- temporary compatibility workarounds
- surprising behaviour
- tradeoffs that cannot be inferred from the code

Do not add comments that merely restate what the code does.

Do not delete useful reasoning comments to meet an arbitrary comment ratio.

## GR-STYLE-007: No Em Dashes

The em dash character `U+2014` is forbidden in:

- source code
- comments
- internationalization strings
- tests
- documentation
- generated copy

Use commas, colons, parentheses, or separate sentences.

## GR-STYLE-008: Structured Logging

Use structured logging instead of `console.*`.

- Main-process modules use `logger` or `createLogger()` from `src/main/logger.ts`.
- Renderer components and hooks use the injected `ILogService`.
- Console output is allowed only inside approved logger transport configuration and mock logging implementations.

Logs must not contain secrets, tokens, passphrases, credential ciphertext, or sensitive user data.

## GR-STYLE-009: Pure Formatting Functions

Formatting helpers remain pure and side-effect-free.

Examples include:

- `timeAgo`
- `timeUntil`
- `commandLine`
- `hostLabel`
- status label mapping
- status colour mapping

Do not introduce network, storage, logging, or mutable global state into formatting modules.

# Change Checklists

## IPC Change Checklist

Any new or modified IPC capability must review and update, where applicable:

- `src/shared/ipc.ts`
- main-process handler
- preload bridge
- real renderer service interface
- real renderer service implementation
- mock renderer service implementation
- service identifier
- dependency-injection registration
- consuming feature, entity, widget, or page
- internationalization messages
- tests
- verification commands

Do not expose a new IPC capability directly to components.

## Credential Change Checklist

Credential-related changes must verify:

- plaintext is never persisted
- environment objects contain references only
- `safeStorage` encryption is used
- there is no plaintext fallback
- serialized output excludes secret fields
- renderer-visible objects contain no secrets or ciphertext
- environment deletion removes owned credential entries
- failed decryption is handled safely
- logs contain no credentials
- authentication failures invalidate stale sessions

## Renderer Feature Checklist

Before adding a renderer feature:

- identify the correct FSD layer
- load `@feature-sliced-design`
- determine whether an existing entity or feature owns the behaviour
- avoid direct bridge access
- inject required services
- add internationalization keys
- keep UI and stateful logic separate when practical
- extract independently testable logic
- provide a mock-compatible implementation
- verify the feature in browser or Electron mode

## Main-Process Networking Checklist

Before adding network behaviour:

- validate the target URL
- keep network access in the main process
- attach authentication only at the main boundary
- define timeouts and cancellation
- return typed errors
- translate user-facing failures
- avoid logging secrets
- handle authentication invalidation
- add tests for pure mapping, validation, or classification logic
- ensure renderer consumers see only the shared response contract

## New Service Checklist

A new renderer service should include:

- a focused interface
- a real implementation
- a mock implementation
- a service identifier
- dependency-injection registration
- typed return values
- translated user-facing errors
- tests for pure logic
- no direct use from components without injection
